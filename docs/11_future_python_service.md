## 一、为什么需要独立服务

### 1.1 当前架构的局限

**Next.js 主应用适合**:
- 用户认证和授权
- 数据库 CRUD 操作
- 页面渲染
- 简单业务逻辑

**Next.js 主应用不适合**:
- CPU 密集型任务（OCR、PDF 解析）
- 长时间运行任务（批量处理）
- 需要特定依赖的任务（Python ML 库）
- 异步后台任务

### 1.2 何时需要拆分

**触发条件**（满足任意一条）:
1. 用户强烈需要 OCR 功能
2. 用户强烈需要 PDF 文本提取
3. 用户强烈需要自动摘要
4. 用户强烈需要智能分类
5. 主应用性能受影响

**不要过早拆分**:
- MVP 阶段不拆分
- 用户需求不明确时不拆分
- 功能可以在主应用中实现时不拆分

### 2.1 核心职责

**内容处理**:
- OCR 图片文字识别
- PDF 文本提取
- HTML 正文解析
- 文本清洗和格式化

**智能分析**:
- 自动分类
- 自动摘要

- 批量处理

### 2.2 不负责什么

**不负责核心业务**:
- ❌ 不管理用户认证
- ❌ 不管理用户权限
- ❌ 不管理订阅关系
- ❌ 不管理收件箱
- ❌ 不管理笔记和专题

**不负责数据存储**:
- ❌ 不直接操作主数据库
- ❌ 不存储用户数据
- ✅ 只处理内容，返回结果给主应用

## 三、技术栈

### 3.1 推荐技术栈

**框架**: Python + FastAPI
- 轻量高性能
- 异步支持
- 自动生成 API 文档
- 类型提示支持

**OCR**: Tesseract + pytesseract
- 开源免费
- 支持多语言
- 准确率较高

**PDF 解析**: PyPDF2 或 pdfplumber
- 提取文本
- 提取元数据
- 处理复杂 PDF

**HTML 解析**: BeautifulSoup4 或 trafilatura
- 提取正文
- 去除广告和导航
- 保留格式

**NLP**: 
- 摘要：sumy 或 transformers
- 分类：scikit-learn 或 transformers
- 关键词：jieba + TF-IDF

**任务队列**: Celery + Redis
- 异步任务
- 定时任务
- 任务重试

### 3.2 不推荐的技术

**重型框架**:
- ❌ Django（过于重型）
- ❌ Flask（不如 FastAPI 现代）

**商业 API**:
- ❌ Google Cloud Vision（成本高）
- ❌ AWS Textract（成本高）
- ✅ 优先使用开源方案

## 四、架构设计

### 4.1 服务架构

```
┌─────────────────────────────────────────┐
│         Next.js 主应用                   │
│  (用户认证、业务逻辑、数据存储)           │
└─────────────┬───────────────────────────┘
              │ HTTP API
              ↓
┌─────────────────────────────────────────┐
│      Python 内容处理服务                 │
│                                          │
│  ┌──────────────┐  ┌──────────────┐    │
│  │  FastAPI     │  │  Celery      │    │
│  │  (同步 API)  │  │  (异步任务)  │    │
│  └──────────────┘  └──────────────┘    │
│                                          │
│  ┌──────────────────────────────────┐  │
│  │  内容处理模块                     │  │
│  │  - OCR                           │  │
│  │  - PDF 解析                      │  │
│  │  - HTML 解析                     │  │
│  │  - 自动摘要                      │  │
│  │  - 智能分类                      │  │
│  └──────────────────────────────────┘  │
└─────────────────────────────────────────┘
              │
              ↓
        ┌──────────┐
        │  Redis   │
        │ (任务队列)│
        └──────────┘
```

### 4.2 通信方式

**同步 API**（快速任务）:
```
Next.js → HTTP POST → Python FastAPI → 返回结果
```

**异步任务**（慢速任务）:
```
Next.js → HTTP POST → Python FastAPI → 创建 Celery 任务 → 返回任务 ID
Next.js → HTTP GET → Python FastAPI → 查询任务状态 → 返回结果
```

### 4.3 数据流

**用户上传图片 OCR**:
1. 用户在 Next.js 上传图片
2. Next.js 保存图片到 Supabase Storage
3. Next.js 调用 Python 服务 OCR API
4. Python 服务下载图片，执行 OCR
5. Python 服务返回识别文本
6. Next.js 保存文本到数据库

**用户上传 PDF 解析**:
1. 用户在 Next.js 上传 PDF
2. Next.js 保存 PDF 到 Supabase Storage
3. Next.js 调用 Python 服务 PDF 解析 API（异步）
4. Python 服务创建 Celery 任务，返回任务 ID
5. Next.js 轮询任务状态
6. Celery 任务完成后，Next.js 获取结果
7. Next.js 保存文本到数据库

## 五、API 设计

### 5.1 OCR 接口

**接口**: `POST /api/ocr`  
**用途**: 图片文字识别

**请求体**:
```json
{
  "imageUrl": "https://...",
  "language": "chi_sim+eng"
}
```

**响应**:
```json
{
  "text": "识别的文字内容",
  "confidence": 0.95
}
```

---

### 5.2 PDF 解析接口

**接口**: `POST /api/pdf/extract`  
**用途**: PDF 文本提取（异步）

**请求体**:
```json
{
  "pdfUrl": "https://...",
  "pages": "1-10"
}
```

**响应**:
```json
{
  "taskId": "uuid",
  "status": "pending"
}
```

**查询任务**: `GET /api/tasks/{taskId}`

**响应**:
```json
{
  "taskId": "uuid",
  "status": "completed",
  "result": {
    "text": "提取的文本",
    "pages": 10
  }
}
```

---

### 5.3 HTML 解析接口

**接口**: `POST /api/html/extract`  
**用途**: 提取网页正文

**请求体**:
```json
{
  "url": "https://...",
  "html": "<html>...</html>"
}
```

**响应**:
```json
{
  "title": "文章标题",
  "author": "作者",
  "content": "正文内容",
  "excerpt": "摘要"
}
```

---

### 5.4 自动摘要接口

**接口**: `POST /api/summarize`  
**用途**: 生成文本摘要

**请求体**:
```json
{
  "text": "长文本内容",
  "maxLength": 200
}
```

**响应**:
```json
{
  "summary": "摘要内容"
}
```

---

### 5.5 智能分类接口

**接口**: `POST /api/classify`  
**用途**: 自动分类文本

**请求体**:
```json
{
  "text": "文本内容",
  "categories": ["技术", "生活", "思考"]
}
```

**响应**:
```json
{
  "category": "技术",
  "confidence": 0.85
}
```

## 六、部署方案

### 6.1 开发环境
- 本地运行 Python 服务
- Next.js 通过 localhost 调用

### 6.2 生产环境

**选项 A: Docker 容器**
- 使用 Docker 打包 Python 服务
- 部署到 Fly.io / Railway / Render
- Next.js 通过内网或公网调用

**选项 B: Serverless**
- 使用 AWS Lambda / Google Cloud Functions
- 适合低频调用场景
- 冷启动可能较慢

**选项 C: 独立服务器**
- 部署到独立服务器
- 适合高频调用场景
- 需要自己维护

### 6.3 推荐方案
- MVP 后初期：Docker + Fly.io（简单便宜）
- 用户量增长后：独立服务器（性能稳定）

## 七、安全考虑

### 7.1 认证授权
- Python 服务不直接面向用户
- 只接受来自 Next.js 的请求
- 使用 API Key 或 JWT 认证

### 7.2 输入验证
- 验证文件类型和大小
- 防止恶意文件上传
- 限制处理时间

### 7.3 资源限制
- 限制并发任务数
- 限制单个任务资源使用
- 防止 DoS 攻击

## 九、何时开始实现

### 9.1 不要过早实现

**不要在以下情况实现**:
- MVP 还未完成
- 用户需求不明确
- 功能可以在主应用中实现
- 用户量很小

### 9.2 何时开始实现

**满足以下条件时考虑实现**:
1. MVP 已完成并稳定运行
2. 用户强烈需要 OCR/PDF 解析功能
3. 有足够的用户量支撑成本
4. 主应用性能受影响

### 9.3 实现步骤

1. **需求确认**（1 天）
   - 确认用户需求
   - 确认功能优先级
   - 确认技术方案

2. **原型开发**（3-5 天）
   - 搭建 FastAPI 框架
   - 实现核心功能
   - 本地测试

3. **集成测试**（2-3 天）
   - Next.js 集成调用
   - 端到端测试
   - 性能测试

4. **部署上线**（2-3 天）
   - 配置生产环境
   - 部署服务
   - 监控和日志

5. **优化迭代**（持续）
   - 根据用户反馈优化
   - 性能优化
   - 功能扩展

## 十、替代方案

### 10.1 使用第三方 API

**优点**:
- 无需开发和维护
- 功能强大准确
- 快速上线

**缺点**:
- 成本较高
- 依赖第三方
- 数据隐私问题

**推荐场景**:
- 快速验证需求
- 用户量小
- 预算充足

### 10.2 在主应用中实现

**优点**:
- 架构简单
- 无需额外服务
- 开发成本低

**缺点**:
- 性能受限
- 功能受限
- 难以扩展

**推荐场景**:
- 功能简单
- 用户量小
- MVP 阶段
