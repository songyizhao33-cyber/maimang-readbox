## 二、Karpathy 四原则

本项目采用 Andrej Karpathy 启发的编码原则：

### 2.1 Think Before Coding（编码前思考）

**不要假设。不要隐藏困惑。呈现权衡。**

在实现前必须：
- 明确说明假设
- 如果有多种解释，列出所有解释
- 如果有更简单的方法，说出来
- 如果不清楚，停下来提问

**示例**:
```
❌ 错误：直接实现订阅功能
✅ 正确：
"订阅功能有两种理解：
1. 订阅后立即收到历史文章
2. 订阅后只收到新文章

我假设是方案 2（只收到新文章），因为这符合邮箱式阅读的定位。
如果需要方案 1，请告知。"
```

### 2.2 Simplicity First（简洁优先）

**用最少的代码解决问题。不要过度推测。**

- 不添加未要求的功能
- 不为单次使用创建抽象
- 不添加未要求的"灵活性"
- 不为不可能场景做错误处理
- 如果 200 行可以写成 50 行，重写

**检验标准**: 资深工程师会觉得这过于复杂吗？

**示例**:
```typescript
// ❌ 过度工程
class ArticleService {
  private repository: IArticleRepository;
  private validator: IArticleValidator;
  private publisher: IArticlePublisher;
  
  constructor(deps: ArticleServiceDependencies) {
    // 10 行依赖注入代码
  }
  
  async publish(id: string): Promise<Result<Article, Error>> {
    // 50 行复杂逻辑
  }
}

// ✅ 简洁实现
export async function publishArticle(id: string) {
  const article = await db.articles.findUnique({ where: { id } });
  if (!article) throw new Error('Article not found');
  
  await db.articles.update({
    where: { id },
    data: { status: 'published', publishedAt: new Date() }
  });
  
  await createInboxItems(article);
  return article;
}
```

### 2.3 Surgical Changes（精准修改）

**只碰必须碰的。只清理自己造成的混乱。**

编辑现有代码时：
- 不"改进"相邻代码
- 不重构没坏的东西
- 匹配现有风格
- 如果注意到无关死代码，提一下但不删除

当你的改动产生孤儿代码时：
- 删除因你的改动而无用的导入/变量/函数
- 不删除预先存在的死代码

**检验标准**: 每一行修改都应该能直接追溯到用户请求。

### 2.4 Goal-Driven Execution（目标驱动执行）

**定义成功标准。循环验证直到达成。**

将任务转化为可验证目标：

| 不要这样做 | 转化为 |
|-----------|--------|
| "添加验证" | "为无效输入编写测试，然后让它们通过" |
| "修复 bug" | "编写重现 bug 的测试，然后让它通过" |
| "重构 X" | "确保重构前后测试都能通过" |

对于多步骤任务，说明简短计划：
```
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
3. [步骤] → 验证: [检查]
```

## 三、每个任务前必须回答的问题

### 3.1 需求理解
1. 这个任务的核心目标是什么？
2. 有哪些假设？
3. 有哪些不确定的地方？
4. 有哪些边界情况？

### 3.2 范围控制
5. 这个任务明确不包含什么？
6. 是否会引入未要求的功能？
7. 是否符合"小而美"定位？
8. 是否违反外部内容合规原则？

### 3.3 技术方案
9. 涉及哪些文件？
10. 涉及哪些数据表？
11. 涉及哪些接口？
12. 是否有更简单的实现方式？

### 3.4 验收标准
13. 如何验证功能正确？
14. 需要运行哪些命令？
15. 需要检查哪些文件？
16. 如何确认没有破坏现有功能？

## 五、Codex 任务格式

每个 Codex 任务必须包含：

### 5.1 任务头部
```markdown
# T10: 实现用户注册登录

**优先级**: 高
**依赖**: 无
**预计时间**: 2-3 小时
```

### 5.2 任务描述
```markdown
## 任务描述

使用 Supabase Auth 实现用户注册登录功能。

包括：
- 邮箱密码注册
- 登录
- 登出
- 注册后自动创建 profiles 记录
```

### 5.3 涉及文件
```markdown
## 涉及文件

- src/app/(auth)/login/page.tsx
- src/app/(auth)/register/page.tsx
- src/app/api/auth/register/route.ts
- src/app/api/auth/login/route.ts
- src/app/api/auth/logout/route.ts
- src/lib/supabase/client.ts
- src/lib/supabase/server.ts
```

### 5.4 验收标准
```markdown
## 验收标准

- [ ] 用户可以通过邮箱密码注册
- [ ] 注册后自动创建 profiles 记录
- [ ] 用户可以登录并获得 JWT token
- [ ] 用户可以登出
- [ ] 登录状态持久化
- [ ] TypeScript 无类型错误
- [ ] ESLint 无警告
```

### 5.5 验证命令
```markdown
## 验证命令

```bash
pnpm lint
pnpm build
pnpm test
```
```

### 5.6 不做什么
```markdown
## 不做什么

- ❌ 不实现第三方登录（Google, GitHub 等）
- ❌ 不实现手机号登录
- ❌ 不实现邮箱验证
- ❌ 不实现密码重置（MVP 阶段）
```

## 七、如何避免过度工程

### 7.1 过度工程的信号

**抽象过度**:
- 为单次使用创建接口
- 为 2-3 个相似函数创建基类
- 引入复杂设计模式

**功能过度**:
- 添加"可能有用"的功能
- 添加"未来可能需要"的配置
- 实现未要求的边缘情况处理

**依赖过度**:
- 引入重型库解决简单问题
- 引入多个功能重叠的库
- 引入未充分评估的新技术

### 7.2 如何避免

**三问法**:
1. 这是当前任务必需的吗？
2. 有更简单的方式吗？
3. 资深工程师会觉得这过于复杂吗？

**延迟决策**:
- 不为未来需求提前设计
- 等到真正需要时再抽象
- 三次重复再考虑抽象

**代码审查**:
- 每次提交前自我审查
- 删除未使用的代码
- 简化过度复杂的逻辑

## 八、常见陷阱和应对

### 8.1 陷阱：默默假设需求

**错误示例**:
```
用户：实现订阅功能
AI：（默默假设需要邮件通知，实现了邮件服务）
```

**正确做法**:
```
用户：实现订阅功能
AI：订阅功能我理解为：
1. 用户点击订阅按钮
2. 创建订阅关系
3. 作者发布新文章时，文章进入订阅者收件箱

不包含：
- 邮件通知
- 推送通知
- RSS 订阅

是否正确？
```

### 8.2 陷阱：顺手重构无关代码

**错误示例**:
```
任务：修复登录 bug
AI：（顺便重构了整个 auth 模块，改了 10 个文件）
```

**正确做法**:
```
任务：修复登录 bug
AI：只修改 login/page.tsx 中的 bug
如果发现其他问题，记录到 TODO 但不修改
```

### 8.3 陷阱：过度抽象

**错误示例**:
```typescript
// 为 2 个相似函数创建复杂抽象
abstract class BaseService<T> {
  abstract validate(data: T): ValidationResult;
  abstract transform(data: T): TransformedData;
  // 10 个抽象方法
}
```

**正确做法**:
```typescript
// 保持简单，等到有 3+ 个相似函数再考虑抽象
export function createArticle(data: ArticleInput) {
  // 直接实现
}

export function createExternalItem(data: ExternalItemInput) {
  // 直接实现
}
```
